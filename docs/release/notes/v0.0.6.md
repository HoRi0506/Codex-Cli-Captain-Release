# v0.0.6-pre

> Pre-release notice: `v0.0.6-pre` is a beta/pre-release train. Install or update, fully restart Codex CLI, then run `ccc check-install`.
> Platform status: macOS remains the primary supported path. Linux and Windows remain early-testing targets unless and until they are verified in real environments.

## Theme

`v0.0.6-pre` focuses on the next step in CCC operating-model clarity.

The release target is to make the launch experience, lane boundaries, and per-subagent output conventions easier to reason about before more runtime behavior lands. The main goal is to separate what the operator sees, what each role owns, and what each lane must emit.

## Planning Targets

- OMA-inspired workflow launch UX that makes starting a task-card feel structured instead of opaque.
- Role/workflow/shared-rule spec separation so operator guidance, workflow wiring, and shared invariants do not blur together.
- Config-backed runtime dispatch so lane selection and execution policy come from config instead of hard-coded branching.
- Per-lane result/log/recap artifact conventions so every lane produces predictable evidence for review and follow-up.
- Verify/retry workflow and recap/reporting pattern so failed or incomplete work can be rerun with a consistent recap trail.
- Per-CCC-subagent ICL and prompt-surface needs so each specialist gets only the instruction surface it actually needs.
- Additional improvement candidates that can be promoted once the core structure is stable.

## Planned Runtime Shape

- Launch UX should guide the operator through a clear workflow entry point, with explicit role selection and a readable handoff into lane execution.
- Role specifications should describe who does what, workflow specs should describe sequencing and transitions, and shared-rule specs should hold invariants that apply to every lane.
- Runtime dispatch should read from config-backed policy so CCC can choose the right lane or specialist without baking the decision tree into release text.
- Each lane exposes a `lane_artifact_contract` for result/log/recap naming so downstream review can find evidence without special-case lookup.
- Verify and retry should be treated as first-class workflow states, with recap and report output that explains what changed, what remained open, and what should happen next.
- Each CCC subagent should expose only the prompt surface and instruction context needed for that role, with no accidental leakage of unrelated operator policy.
- Captain may directly finish trivial or fallback work, but if that work reveals non-trivial follow-up, captain should stop expanding locally and delegate the follow-up to the appropriate CCC subagent.

## Subagent ICL Targets

These are prompt-surface planning targets, not claims that current specialists are incapable.

- `ccc_tactician`: add stronger planning context for alternatives, assumptions, gates, parallelization fit, and when to ask the operator for a decision.
- `ccc_scout`: keep the read-only evidence contract, but add light conventions for source prioritization, multi-source summaries, and result artifacts.
- `ccc_raider`: add a scoped mutation checklist covering boundary preservation, test selection, validation evidence, and repair handoff when implementation fails.
- `ccc_scribe`: add light guidance for release-note style, operator-facing tone, translation fidelity, and structured reporting.
- `ccc_arbiter`: add review rubrics for regression risk, security, performance, accessibility, test gaps, and file-line remediation output.
- `ccc_sentinel`: add a routing and ownership-classification playbook so path ownership, lane selection, and shared-scope conflicts are judged consistently.
- `ccc_companion_reader`: keep the current read-only role mostly intact, with optional result/log artifact conventions if CCC adopts per-lane artifact paths.
- `ccc_companion_operator`: add dedicated guidance for git, `gh`, release operations, commit boundaries, and operator-side mutation safety.

## Additional Improvement Candidates

These are follow-up candidates for after the core `0.0.6-pre` slices stabilize. They are not claims about current runtime behavior.

### P0: Operator Clarity

- A clearer operator-facing launch summary for complex multi-lane runs.
- Better feedback when config dispatch and requested workflow shape do not match.

### P1: Evidence and Replayability

- Stronger artifact naming for lane outputs, retries, and final recap bundles.
- More explicit reporting for recovered retries versus clean first-pass completion.

### P2: Specialist Surface and Doc Boundaries

- Narrower specialist prompt templates for docs, scout, raider, arbiter, and companion flows.
- A tighter public/private split between operator-facing docs and internal lane policy details.

## Planning Ledger

- [x] OMA-inspired workflow launch UX: compact and text status now surface the launch lane(s) and expected fan-in, while task-card root fields carry role, agent, and scope without duplicating them inside `launch_visibility`.
- [x] Role/workflow/shared-rule spec separation: delegation plans now carry `spec_surfaces`, compact status preserves the split, and text status renders it as `Spec Split`.
- [x] Config-backed runtime dispatch: delegation plans now carry a structured `runtime_dispatch` snapshot, compact status projects it, and text status renders one concise source-attributed `Dispatch` line.
- [x] Per-lane result/log artifact conventions: status and delegation payloads now expose a shared `lane_artifact_contract` for result, log, and recap naming.
- [x] Verify/retry workflow and recap/reporting pattern: delegation plans now carry a `verify_retry_recap_report_contract`, compact status projects it, and text status renders a concise `Verify/Retry/Recap/Report` line.
- [x] Per-CCC-subagent ICL/prompt-surface needs: first custom-agent instruction slice landed for captain, tactician, scout, raider, scribe, arbiter, sentinel, companion reader, and companion operator.
- [x] Additional improvement candidates refinement

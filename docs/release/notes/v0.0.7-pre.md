# v0.0.7-pre

> Pre-release note: `v0.0.7-pre` keeps the public docs aligned while the next presentation slice lands. The emphasis is on making the README positioning copy easier to notice, keeping the 0.0.7 plan smoke-backed, and preserving a clear operator-facing paper trail.

## Theme

`0.0.7-pre` is a documentation-presentation and operator-clarity slice with a runtime repair for LongWay checklist visibility and specialist routing.

The immediate goal is to keep the README positioning copy at the top of the source and release README files, but make that copy stand out more clearly in English, Korean, and Japanese with simple Markdown formatting. The release record keeps the smoke evidence as history, then separates the runtime slices that have already landed from the remaining 0.0.7 documentation and presentation work.

## Historical Smoke Baseline

Companion operator smoke already covers the core route and operator-text surfaces:

- `check-install` passed.
- `way` routed to `ccc_tactician`.
- `explore` routed to `ccc_scout`.
- `git commit` routed to `ccc_companion_operator`.
- code mutation routed to `ccc_raider`.
- docs/operator-text previously routed to `ccc_raider` instead of `ccc_scribe`.
- `ccc start` previously did not show the selected route or agent until `ccc status`.
- `ccc status` previously showed `Current Item` but not a checklist breakdown.

## Landed Runtime Fixes

- docs/operator-text routing now prefers `ccc_scribe`.
- `ccc start` now shows the selected route and agent at launch time.
- `ccc status` now shows a checklist-style completed/current/remaining breakdown.
- `ccc checklist --text` now provides a separate checklist-only surface: a `LongWay` header plus separate `[x]` / `[>]` / `[ ]` item rows for a run.
- terminal `ccc subagent-update --text` now persists the active LongWay phase completion before rendering status, so the completion output prints the checked checklist item.
- multi-item LongWay completion now advances the active item in order instead of completing the last pending item, and checklist labels prefer persisted task titles before falling back to internal phase names.
- the packaged `$cap` skill now requires `ccc checklist --text` to be shown as a separate operator-visible lifecycle block immediately after run start, after each terminal fan-in, after the following `merged` update, and before the next task/replan/orchestration step.
- code/config repair requests with incidental git or release context stay on `ccc_raider`; explicit operator-side git/gh release mutations still route to `ccc_companion_operator`.
- stale generated defaults now upgrade the `code specialist`/raider role from `gpt-5.5` `medium` to `gpt-5.5` `high`, matching the recommended role table.
